The 0-1 Knapsack Problem

Given n items with weights wi and values vi, and a knapsack of capacity C — pick a subset that maximizes total value without exceeding capacity.

maximize   Σ vᵢ · xᵢ
subject to Σ wᵢ · xᵢ ≤ C
           xᵢ ∈ {0, 1}

Why it matters

  • Canonical NP-hard problem
  • Same family as driver-to-region, truck loading, budget allocation
  • Solvers like PuLP/CBC use B&B under the hood
Quick problem framing. Every x_i is a binary in-or-out choice. The constraint is capacity. The objective is total value. NP-hard means no known polynomial algorithm that's always exact. Connect it to logistics: items become shipments and capacity becomes the truck. The driver-to-region assignment in my project is the same family — binary decisions with linear constraints.

Baseline: Brute Force

Enumerate every subset — 2n of them.

for mask in range(1 << n):           # 2^n iterations
    w, v = 0, 0
    for i in range(n):
        if mask & (1 << i):
            w += weights[i]
            v += values[i]
    if w <= capacity and v > best:
        best = v

Exponential. Adding one item DOUBLES the work. Even in C++ at n=25 we're at 15 seconds. At n=30 you'd be at 8 minutes. At n=40, you'd be measuring in days. This is the wall we have to break.

Branch & Bound — The Idea

  1. Branch. At each item, recurse on include vs exclude.
  2. Bound. Compute an upper bound on the best the subtree could produce.
  3. Prune. If bound ≤ current best, skip the whole subtree.

Items are pre-sorted by value / weight descending, so the include branch is always the greedy guess — it finds a strong incumbent fast, which makes the bound tighter and prunes harder.

The whole idea is: never expand a subtree we can prove won't beat what we already have. Three parts. Branch = the decision tree. Bound = an optimistic estimate of what's possible from here. Prune = cut whole subtrees. The sorting is the trick — it makes the greedy choice the first thing we try, so we have a strong incumbent quickly, which makes the bound condition fire more often.

B&B Decision Tree — On Our Tiny Instance

Out of 16 possible subsets, B&B visits 9 nodes — 3 pruned by bound, 2 by infeasibility. The best feasible solution B+C = 22 is found early and powers the rest of the pruning.

Walk left to right. We try include-A first (greedy), then include-B, hit a value of 20 quickly. Then under A=0, B=1, C=1 we find value 22 — that's our incumbent. Now look at the red nodes: they're not explored because their LP-relaxation upper bound is already ≤ 22, so there's no way they can beat it. The orange nodes are infeasible — over capacity. We've reduced 16 subsets to 9 explored nodes. At n=25 the reduction is closer to a million-to-one. Next slide.

The Bound: LP Relaxation

Relax xi ∈ {0,1} → xi ∈ [0,1]. The fractional knapsack has a closed-form greedy answer.

def upper_bound(level, cur_weight, cur_value):
    bound = cur_value
    rem = capacity - cur_weight
    for i in range(level, n):
        if w[i] <= rem:           # take the whole item
            bound += v[i]
            rem -= w[i]
        else:                      # take a fraction of the next item
            bound += v[i] * rem / w[i]
            break
    return bound

Items already sorted by value/weight, so this greedy fill is the LP optimum. Floor of the bound is still valid for integers — that's why ASM can do this in pure integer math, no FPU.

Why this bound? Because it's tight AND fast. Tight: the LP relaxation is the tightest polynomial bound we know. Fast: with sorted items, it's a single O(n) sweep. The integer-floor trick is what lets the ASM version stay in integer registers — no floating point at all.

How Much Does Pruning Actually Save?

At n=25, the brute force search has 33 million subsets; B&B visits only a few dozen. That's less than 0.0001% of the tree.

This is the punchline of the bound. Left chart: brute force in red versus what B&B actually touches in green, log scale. The gap is enormous and grows with n. Right chart: the same data expressed as a percentage. At n=25 we touch under a thousandth of one percent of the search space. THIS is why B&B works at scale.

Same Algorithm, Three Languages

void branch(int level, int cw, int cv) {
    if (cv > best) { best = cv; ... }
    if (level == N) return;
    if (upper_bound(level, cw, cv) <= best) return;   // PRUNE

    if (cw + W[level] <= CAP)                          // INCLUDE
        branch(level+1, cw + W[level], cv + V[level]);
    branch(level+1, cw, cv);                            // EXCLUDE
}

Python and C++ are line-for-line equivalent. The x86-64 ASM uses the Win64 ABI and recurses via call/ret — r12/r13 hold the sorted arrays, ebx holds the running incumbent, the bound is computed inline in pure integer math.

Same algorithm, same shape in all three languages. That's deliberate — it lets us isolate the language overhead from the algorithm overhead. The ASM version was the most fun: I used callee-saved registers as "globals" for the search state, and recursed using the real call stack. The bound computation is all integer division.

Algorithmic Win: B&B vs Brute Force

Python at n=25: 102 s → 96 μs — over 1,000,000× faster. Same answer (1153). Same machine. Just a better algorithm.

This is the big number. A million times faster, just from a smarter algorithm — no change in hardware, no change in language. The languages all benefit, because the algorithm wins are independent of the language wins. Note the log scale.

3-Way B&B: Python vs C++ vs ASM

Across n = 3 … 25, C++ averages ~13× faster than Python; hand-tuned x86-64 ASM averages ~52× over Python and ~3× over C++.

Now within B&B, look at language. The relative ranking is consistent: ASM beats C++ beats Python. ASM's edge over C++ is modest — ~3× — and comes from avoiding the function-call overhead and keeping the search state in registers. Whether that's worth maintaining hand-written assembly is a separate question — but it cleanly quantifies the ceiling.

The lesson: algorithmic choice dominates language choice. A million× win from B&B; a 50× win from ASM. Both matter — but you can't out-engineer an exponential.

Here's the bridge to the capstone. The driver-to-region problem is an integer linear program — binary x[d,r], cost-minimizing objective, eligibility and coverage constraints. When I write prob.solve() in PuLP, it hands the problem to CBC. CBC IS Branch and Bound — with cuts and heuristics on top, but the spine is B&B. On the LLM track, knapsack is the right testbed because it's small, well-defined, and we have ground truth from this work. If I ask an LLM to write a knapsack solver, I can measure whether it found B&B or just brute force.

Takeaways

  • Brute force is a useful baseline — and a clear demonstration of why we need smarter algorithms.
  • B&B uses an LP-relaxation bound to prune the search tree: correct, optimal, dramatically faster.
  • Same algorithm in 3 languages isolates language overhead from algorithm overhead — the speedups multiply independently.
  • Connection: the ILP solvers I use for logistics assignment are B&B under the hood. Understanding B&B = understanding the tools.
