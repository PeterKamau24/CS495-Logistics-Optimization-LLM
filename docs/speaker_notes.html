Slide 2The 0-1 Knapsack Problem
On screen: Math formulation + “why it matters” bullets.

Quick problem framing. Every x_i is a binary in-or-out choice. The constraint is capacity. The objective is total value.

NP-hard means no known polynomial algorithm that's always exact.

Connect it to logistics: items become shipments and capacity becomes the truck. The driver-to-region assignment in my project is the same family — binary decisions with linear constraints.
Slide 3A Concrete Tiny Instance
On screen: 4 items A–D with values, weights, and value/weight ratios.

Anchor instance. The ratio column is the LP “bang for your buck” — high ratio items are the greedy first choices.

Optimal here is B+C = value 22 with weight 8. Hold that answer for the tree slide coming up.

Slide 4Baseline: Brute Force
On screen: Python loop enumerating 2^n subsets.

Exponential. Adding one item doubles the work.

Even in C++ at n=25 we're at 15 seconds. At n=30 you'd be at 8 minutes. At n=40, days. This is the wall we have to break.

Slide 5Brute Force — Python vs C++ vs Assembly
On screen: Bar chart, log scale, three languages from n=3 to n=25.

This is the brute-force benchmark on its own. Note the log scale — every step up in n is a noticeable jump.

Assembly is the fastest by a constant factor, but the curve still explodes.

Takeaway: no amount of low-level optimization beats the exponential. We need a smarter algorithm.
Slide 6Exponential Scaling — Quantified
On screen: 3-panel chart — curve fits (left), growth rate (middle), performance gap (right).

This slide is the rigor check on the brute-force claim.

Left panel: I fit T = a · 2^n to each language's data — the fits are nearly perfect, confirming the asymptotic.

Middle panel: between consecutive instance sizes, the runtime ratio sits right on the theoretical doubling line.

Right panel: the speedup gap between Python and the compiled languages isn't constant — it grows, because a constant-factor advantage applied to exponential work shows up more dramatically as n increases.

This is the empirical case for “you can't out-engineer 2n.”
Slide 7Branch & Bound — The Idea
On screen: Three numbered steps — Branch / Bound / Prune.

The whole idea: never expand a subtree we can prove won't beat what we already have.

Three parts. Branch = the decision tree. Bound = an optimistic estimate of what's possible from here. Prune = cut whole subtrees.

The sorting is the trick — it makes the greedy choice the first thing we try, so we have a strong incumbent quickly, which makes the bound condition fire more often.

Slide 8B&B Decision Tree — On Our Tiny Instance
On screen: Annotated decision tree with INCLUDE/EXCLUDE branches, pruned nodes in red, infeasible in orange.

Walk the tree left to right. We try include-A first (greedy), then include-B, hit a value of 20 quickly.

Then under A=0, B=1, C=1 we find value 22 — that's our incumbent.

Now look at the red nodes: they're not explored because their LP-relaxation upper bound is already ≤ 22, so there's no way they can beat it. The orange nodes are infeasible — over capacity.

We've reduced 16 subsets to 9 explored nodes. At n=25 the reduction is closer to a million-to-one.
Slide 9The Bound: LP Relaxation
On screen: upper_bound function in Python.

Why this bound? Because it's tight AND fast.

Tight: the LP relaxation is the tightest polynomial bound we know.

Fast: with sorted items, it's a single O(n) sweep.

The integer-floor trick is what lets the ASM version stay in integer registers — no floating point at all.

Slide 10Same Algorithm, Three Languages
On screen: C++ branch() function showing the same shape as the Python version.

Same algorithm, same shape in all three languages. That's deliberate — it lets us isolate the language overhead from the algorithm overhead.

The ASM version was the most fun: I used callee-saved registers as “globals” for the search state, and recursed using the real call stack. The bound computation is all integer division.

Slide 12Connection to My Capstone
On screen: Two columns — Logistics ILP track, LLM track.

Here's the bridge to the capstone. The driver-to-region problem is an integer linear program — binary x[d,r], cost-minimizing objective, eligibility and coverage constraints.

When I write prob.solve() in PuLP, it hands the problem to CBC. CBC IS Branch and Bound — with cuts and heuristics on top, but the spine is B&B.

On the LLM track, knapsack is the right testbed because it's small, well-defined, and we have ground truth from this work. If I ask an LLM to write a knapsack solver, I can measure whether it found B&B or just brute force.

Slide 15Are the Solvers Actually Correct?
On screen: Four verification angles + passing pytest output.

A benchmark is only meaningful if the solver is correct. Four solvers, four angles of verification.

Brute force is the oracle because it tries every subset — it cannot be wrong on small n. So if B&B agrees with brute force on 50 random seeded instances, plus the canonical instance, plus the disaster-relief instance, that's strong evidence the pruning logic is sound.

The invariant check also catches the subtler bug class where the value is right but the reported item list is wrong.

Nine tests pass in under a second — full suite runs on every change.
Slide 16Takeaways & Q&A
On screen: Five takeaway bullets + “Questions?”

Close with the takeaways. Brute force is the baseline that makes the win measurable. B&B uses a clever bound to prune. Same algorithm in three languages cleanly separates the two dimensions. And the connection — this is the engine of CBC, which is the engine of my capstone.

Open the floor.

Likely Questions — prepared answers

  • “Why not dynamic programming?” — DP is pseudo-polynomial in capacity, great when capacity is small, but for the LP–ILP connection B&B is the right thing to study.
  • “Does the LLM track use this?” — yes, for evaluation: same instances, compare LLM-generated solvers against this baseline.
  • “Why ASM?” — partly to quantify the ceiling, partly because the capstone has a language-comparison angle.
  • “How do you know the solvers are correct?” — cross-checked against brute force (the oracle) on 50 random seeded instances plus two known-answer instances. Full test suite is in tests/test_knapsack.py.
